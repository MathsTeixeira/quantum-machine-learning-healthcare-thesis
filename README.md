## Repository scope

This repository contains the practical implementation associated with the dissertation *Systematic Review of Quantum Machine Learning for Healthcare and a Practical Intervention for Diagnostic Classification*.

The work combines a PRISMA-aligned systematic review of quantum machine learning in biomedical and digital health with a diagnostic intervention for breast cancer histopathology classification from the BreaKHis dataset. The intervention employs a hybrid pipeline with classical feature extraction and compression followed by a quantum support vector machine and evaluation by accuracy, precision, recall, specificity, F1-score, MCC, and AUC-ROC.

The dissertation PDF and the BreaKHis dataset are not included in this repository.

## Notebooks

### `QML_Healthcare_Simulator_Final.ipynb`

This notebook contains the final simulator / analytic statevector-equivalent implementation of the dissertation’s practical diagnostic intervention for breast cancer histopathology classification.

The workflow uses the BreaKHis dataset with patient-wise train/validation/test splitting to reduce leakage. A frozen pretrained ResNet-50 extracts image representations, and a lightweight projection head compresses each image into a scalar parameter θ. The scalar θ is normalized using training-only statistics and used as input to a one-qubit quantum feature map H → Ry(θπ/2). The resulting quantum fidelity kernel is computed analytically, equivalent to noiseless statevector simulation, and passed to a classical SVM solver as a precomputed kernel.

IBM Quantum hardware execution is not performed in this notebook. The hardware status is explicitly logged as disabled, and the main result should be interpreted as simulator-equivalent.

### `QML_Healthcare_IBM_Hardware_Minimal.ipynb`

This notebook contains the IBM Quantum hardware feasibility component of the dissertation’s diagnostic intervention. It should be read together with `QML_Healthcare_Simulator_Final.ipynb`, which remains the full diagnostic benchmark.

The hardware workflow does not retrain the ResNet feature extractor and does not run the complete BreaKHis QSVM kernel on quantum hardware. Instead, it loads the normalized scalar θ embeddings generated by the simulator pipeline, selects a balanced subset, builds one-qubit overlap circuits, estimates the quantum kernel on an IBM backend, compares the hardware kernel with the analytic one-qubit kernel, and evaluates a precomputed-kernel SVM on the subset.

The goal is hardware executability and kernel-noise characterization under NISQ constraints. The result should not be interpreted as full hardware diagnostic performance or as evidence of quantum advantage.

## Environment setup

The notebooks were separated into two environments: one for the simulator and one for IBM hardware execution.


## Execution order

### 1. Run the simulator notebook

Run `QML_Healthcare_Simulator_Final.ipynb` first. This notebook performs:

1. environment validation;
2. BreaKHis metadata construction;
3. patient-wise train/validation/test partitioning;
4. ResNet-based scalar θ extraction;
5. training-only normalization of θ;
6. classical SVM baselines;
7. analytic one-qubit quantum kernel construction;
8. QSVM training and evaluation;
9. artifact and manifest saving.

The simulator notebook creates the output directory used by the hardware notebook.

### 2. Run the IBM hardware notebook

Run `QML_Healthcare_IBM_Hardware_Minimal.ipynb` only after the simulator artifacts have been created.

The hardware notebook performs:

1. environment validation;
2. loading of simulator artifacts;
3. balanced subset selection;
4. analytic subset-kernel construction;
5. one-qubit hardware circuit construction;
6. IBM Quantum authentication;
7. backend and qubit selection;
8. circuit transpilation;
9. hardware submission;
10. count extraction;
11. raw and readout-corrected kernel construction;
12. hardware-vs-analytic kernel comparison;
13. subset QSVM evaluation.

Cells that submit IBM Quantum jobs consume QPU runtime. These cells should be run only when IBM Quantum execution is intended.

### Scalar-θ SVM comparison

| Model | Accuracy | Balanced accuracy | Precision | Recall | Specificity | F1-score | MCC | AUC-ROC |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| Classical SVM (linear) | 0.7735 | 0.7661 | 0.9045 | 0.7809 | 0.7513 | 0.8382 | 0.4820 | 0.8038 |
| Classical SVM (RBF) | 0.7968 | 0.7781 | 0.9047 | 0.8153 | 0.7409 | 0.8577 | 0.5143 | 0.8166 |
| QSVM | 0.8000 | 0.7803 | 0.9051 | 0.8196 | 0.7409 | 0.8602 | 0.5196 | 0.8038 |

The QSVM produced the strongest overall balance among the scalar-θ SVM variants, with the highest accuracy, balanced accuracy, F1-score, and MCC. The improvement over the classical RBF SVM baseline is modest, and the RBF baseline has the highest AUC-ROC. The result should therefore be interpreted as evidence of feasibility and reproducibility of the proposed compact QSVM workflow, not as evidence of quantum advantage.

## Reproducibility notes

The notebooks set a fixed seed where applicable and save intermediate artifacts and runtime manifests. The simulator notebook should be considered the primary diagnostic benchmark. The hardware notebook should be considered a reduced hardware feasibility experiment.

Before publishing notebook outputs, clear outputs from cells that contain IBM Quantum account metadata, especially the authentication cell. Do not commit API keys, `.env` files, IBM credential files, datasets, generated outputs, or model checkpoints.
